换句话说，知识越成体系、越能相互印证，它的质量就越高，就越能抵抗“夹生饭”的侵蚀。

# AGE Lite：从重模板到轻骨架的设计思路

AGE Lite 不是把 AGE 简单删薄一层，而是把原本偏重的治理模板，压缩成一套低摩擦、低上下文成本、又能继续长大的工作骨架。

它要解决的核心问题不是“目录太多”，而是 AI 容易一上来就读很多文档、扫很多代码、补很多礼貌性上下文，最后让小任务也像大项目治理。

## 一句话定位

- 强骨架，弱内部结构
- 输入驱动路由，不是固定阅读链
- 保留系统性治理能力，但不强迫每次都走完整治理链

## 吸引子思想

这套模板的核心不是把所有信息都装进文档，而是给项目保留一个稳定吸引子，让系统围绕少量已确认的真相持续收敛。

这里的吸引子不是“长篇目标文档”，而是下面这组约束：

- 用户当前输入优先
- live repo 决定当前实现实际如何工作
- `baseline/` 承载当前已确认的项目真相和长期收敛方向
- `plans/` 与 `logs/` 只记录执行轨迹，不反过来定义系统
- `input/` 和 `references/` 保留来源材料，但不伪装成项目结论

这意味着项目一开始可以不完整，也可以边做边收敛；但未确认的内容必须留在输入层或参考层，只有真正稳定下来的结构性判断，才进入 `baseline/`。

## 为什么要做目录简化

AGE 从重到轻，依据不是“越少越好”，而是只保留真正稳定、长期有价值的边界。

保留下来的，是 6 个角色明确、很难被替代的层：

- live repo
  - 真实代码和真实文件系统本体
- `docs/context/`
  - AI 如何进入项目、如何搜索、何时停下
- `docs/baseline/`
  - 当前真相与长期收敛方向
- `docs/plans/`
  - 复杂任务如何收口
- `docs/logs/`
  - 已落地事实
- `docs/input/` / `docs/references/`
  - 原始输入与外部约束

被拿掉或弱化的，是这些高摩擦习惯：

- 默认固定阅读链
- 每轮都读一圈 `context/baseline/plans/logs`
- 小任务默认补完整文档上下文
- 只要改动就自动写日志
- 预建大量目录，靠结构“看起来完整”

简化的标准只有一个：  
如果某一层不能稳定回答一种不同的问题，它就不该单独存在。

## 这套骨架真正固定的东西

### 1. 强骨架

顶层职责固定：

- `context` 解决“AI 怎么进来”
- `baseline` 解决“项目现在认可什么”
- `plans/logs` 解决“这一轮怎么收口、发生了什么”
- `input/references` 解决“来源材料是什么”

### 2. 弱内部结构

目录内部默认从简：

- 不预设大量子目录
- 先靠 `index.md` 做导航和局部说明
- 只有当内容真实增长到值得拆分时，再细分

这样做的目的，是让模板适配小项目、探索期项目和逐步长大的项目，而不是一开始就要求完整治理外观。

## 系统基础

这套系统的基础不是文档本身，而是三层现实分工：

### 1. 当前输入

用户这次到底要什么，始终优先于文档体系。  
如果用户已经给出明确文件、函数、报错、页面、样式或目录，默认直接去目标，而不是先回头读整套文档。

### 2. 当前真相

系统“应向哪里收敛”的问题，由 `baseline/` 回答。  
系统“现在实际怎么工作”的问题，由 live repo 回答。  
这两者不能混。

### 3. 当前轨迹

复杂任务如何收口，看 `plans/`。

## 维护为什么也要变轻

- 确认痕迹默认留在对话中，不额外生成状态文件

# 一、学习循环的定义

一次学习循环不是“看完一篇文章”，而是：

> **围绕一个当前台阶，完成学习、练习、验证、裁决、维护。**

也就是：

```text
输入材料 / 当前卡点
  ↓
定位当前能力
  ↓
生成当前台阶
  ↓
学习与练习
  ↓
验证
  ↓
用户裁决
  ↓
按需维护文档
```

核心是：**每轮只推进一个主要能力增量。**

# 二、完整工作流

## 1. 输入进入

用户给出：

```text
我想学什么
我卡在哪里
我遇到了什么真实问题
我提供了什么材料
```

这些内容先进入 `input/current.md`，只作为本轮入口。

规则：

```text
input/current.md 只放本轮材料。
本轮结束后必须清空、转移、摘录或废弃。
input 不做长期资料库。
```

---

AI 先看：

```text
baseline/index.md
steps/current.md
input/current.md
```

然后帮用户判断：

```text
当前已确认能力是什么
当前不稳定点是什么
这次输入暴露了什么缺口
```

但这里 AI 只能给**候选判断**，不能直接宣布用户掌握或没掌握。

---

每个台阶必须写成这个结构：

```text
当前能力：
新增能力：
前置知识：
验证方式：
失败回退点：
风险说明：
```

台阶生成规则：

```text
从当前真实能力出发
一次只新增一个主要能力
必须略高于当前能力
必须可测试
必须有失败回退点
```

禁止这种台阶：

```text

学习数据库
学习 TCP
学习权限系统
```

必须改成这种：

```text
能解释一次 HTTP 请求中 IP、端口、TCP 连接分别解决什么问题，

```

AI 可以做：

```text
解释概念
拆前置知识
举例子
生成练习
生成反例
追问漏洞
整理候选理解
```

用户必须做：

```text
自己复述
自己推导
自己写代码
自己判断
自己决定是否通过
```

核心边界：

> **AI 可以辅助学习，但不能替用户完成理解裁决。**

| 验证类型 | 用途 |
|---|---|
| 复述验证 | 防止只会背术语 |
| 无参考复现 | 防止看着会、离开不会 |
| 变形题验证 | 防止只会原题 |
| 反例/边界验证 | 防止理解过度简化 |

```text
暴露问题：

后续处理：
```

| 状态 | 含义 |

# 三、学习循环完成后的维护规则

## 总原则

维护不是每轮自动大改文档，而是：

> **学习完成后，AI 先给维护建议；用户确认后，才写入长期文档。**

维护分 5 类。

---

## 1. `baseline/` 维护规则

`baseline/` 只记录：

```text
已确认能力真相
长期收敛方向
稳定的核心概念理解
已确认的能力边界
反复验证过的关键判断
仍不稳定但重要的薄弱点
```

可以写入 baseline 的条件：

```text
用户完成验证
用户明确确认
该内容不是单次材料摘录
该内容对长期能力结构有价值
```

不能写入 baseline 的内容：

```text
AI 单方面总结
用户还没验证的理解
文章原文摘录
课程笔记堆积
临时计划
一次性情绪判断
带风险推进的内容
```

一句话：

> **baseline 只收“确认后的能力真相”，不收“看起来有道理的材料”。**

---

## 2. `steps/` 维护规则

`steps/` 记录当前台阶和后续台阶。

每轮结束后按状态处理：

| 状态 | steps 处理 |
|---|---|
| 通过 | 当前台阶标记完成，生成下一候选台阶 |
| 失败回退 | 当前台阶保留，新增最近前置台阶 |
| 带风险推进 | 当前台阶标记风险，允许生成下一台阶，但风险必须显式保留 |
| 暂停 | 当前台阶保留，不生成新台阶 |

`steps/current.md` 建议保持这个结构：

```text
当前台阶：
当前能力：
新增能力：
前置知识：
验证方式：
失败回退点：
当前状态：
风险说明：
下一候选台阶：
```

---

## 3. `logs/` 维护规则

`logs/` 记录发生过什么，尤其是验证结果。

每轮学习结束后，建议至少写一条日志：

```text
日期：
本轮台阶：
输入材料：
学习动作：
验证方式：
验证结果：
暴露问题：
用户裁决：
维护结果：
```

`logs/` 不是结论层。

它可以记录：

```text
失败
误解
反复卡点
带风险推进
AI 候选判断
用户裁决过程
```

但不能反过来定义 baseline。

一句话：

> **logs 记录轨迹，baseline 记录确认后的真相。**

---

## 4. `references/` 维护规则

`references/` 放长期可引用材料。

可以放：

```text
文章链接
官方文档链接
教材章节
课程链接
优秀讨论摘录
AI 对话摘录
```

但必须标状态：

```text
未处理
已摘录
已用于某台阶
已验证部分内容
仅供参考
```

重要规则：

> **references 可长期保留，但永远不等于已确认。**

不要把 references 做成大而全资料库。只收对当前能力地图有用的资料。

---

## 5. `input/` 维护规则

`input/` 是临时入口。

每轮结束必须处理：

| 情况 | 处理 |
|---|---|
| 有长期参考价值 | 移到 `references/` |
| 形成确认理解 | 摘要进入 `baseline/` |
| 形成台阶 | 写入 `steps/` |
| 只是过程材料 | 摘要进入 `logs/` |
| 没价值 | 删除 |

规则：

```text
input/current.md 不允许长期堆积。
本轮结束后必须清空或收口。
```

# 四、AI 在工作流里的权限边界

## AI 可以做

```text
读取当前输入
整理候选台阶
指出前置依赖
生成验证题
追问理解漏洞
整理维护建议
起草 baseline 候选内容
```

## AI 不能做

```text
替用户判断已经掌握
未经确认写入 baseline
把 references 当成 baseline
把带风险推进写成已掌握
验证失败后继续硬推后续内容
把大目标直接拆成一堆课程目录
```

## 最关键规则

> **AI 负责提出候选、证据、风险；用户负责裁决。**

# 五、一次完整运行示例

```text

前置知识：
```

| 长文本难写 | 概念解释、验证证据、反思不自然 |
| 容易工具化过重 | 很快变成“做系统”而不是“辅助学习” |

你的核心原则是**脑子不能交出去**，所以文件必须让人容易看、容易审、容易改。纯 JSON 不适合做主源。

## 为什么纯 MD 也不够

纯 MD 很适合写说明，但维护关系会弱：

```md
前置：HTTP、Header、Cookie
相关：Token、CORS、安全整改
```

这样人看得懂，但机器很难稳定检查：

- 节点名是否写错；
- 前置节点是否存在；
- 关系是否断了；
- 状态是否规范；
- 有没有循环依赖；
- 哪些节点无人引用。

所以纯 MD 容易变成“看着有结构，其实关系不可控”。

## 第一版推荐

用这种形式：

```md
---
id: auth-token-session-redis
type: capability-node
status: unstable
prerequisites:
  - http-stateless
  - http-header-cookie
  - server-side-state
used_in:
  - interface-debugging
  - backend-auth
  - security-baseline
risk:
  - token/session/redis-login-state-boundary
---

# 区分 token、session、Redis 登录态

## 能力说明

能解释三者分别保存什么、校验发生在哪里、退出登录如何失效、泄露后风险有什么不同。

## 已验证证据

- 能说明 token 是凭证，不是安全本身。
- 能说明后端必须校验 token。

## 薄弱点

- session 与 Redis 登录态边界容易混。
- token 自包含与服务端登录态容易混。

## 推荐验证方式

- 无参考画出三种登录流程。
- 解释退出登录时三者如何处理。
- 解释 token 被盗后的风险差异。
```

也就是：

> **YAML Front Matter 管关系，Markdown 正文管理解。**

## 当前建议先定一条规则

> **关系字段结构化，解释内容 Markdown 化。**

这样兼顾三点：

| 目标 | 方案 |
|---|---|
| 人能审核 | Markdown 正文 |
| AI 能维护 | YAML 关系字段 |
| 关系不容易断 | 用固定 `id` 做引用，不靠标题文本 |

当前不建议上 JSON 做主文件。后续如果真的要可视化、检查关系，再让脚本从 MD 的 Front Matter 抽取成 JSON。

对，这块应该明确交给 AI 维护。

否则你自己维护 `id / prerequisites / used_in / chains` 会非常痛苦，也违背这个系统的目的。

## 建议分工

| 内容 | 谁维护 | 原因 |
|---|---|---|
| `id` 命名 | AI 起草，人必要时改 | 人手工统一命名很累 |
| `prerequisites` 前置关系 | AI 维护，人审核关键前置 | 关系多，适合 AI 梳理 |
| `used_in` 应用场景 | AI 维护 | 容易遗漏，AI 更适合补全 |
| `related` 相关节点 | AI 维护 | 主要用于导航和检索 |
| `chains` 能力链 | AI 起草和更新，人确认主链是否合理 | 主链影响学习方向，不能完全交出去 |
| `status` 状态 | 人裁决，AI 只能建议 | 是否掌握必须由人判断 |
| `baseline` 写入 | 人确认后 AI 执行 | 防止 AI 把候选写成真相 |

## 核心规则

可以定成：

> **结构维护交给 AI，状态裁决交给人。**

也就是：

```text
AI 维护：
- 节点关系
- 链接补全
- 引用修正
- 缺失检查
- 重复节点合并建议
- 关系断裂检查

人裁决：
- 这个节点是否真的掌握
- 这个前置是否关键
- 这条能力链是否符合真实工作
- 是否允许进入 baseline
```

## 最重要的限制

AI 可以自动改关系，但改完必须汇报：

```text

新增了哪些关系
删除了哪些关系
哪些判断需要用户确认
哪些内容不能进入 baseline
```

所以这块不是“人手工维护图”，而是：

> **AI 做图谱维护员，人做路线裁判。**

## 1. AI 维护关系时要分级

不能所有关系都同等重要。

| 关系类型 | 维护方式 |
|---|---|
| `prerequisites` 前置关系 | AI 可建议，人最好确认 |
| `used_in` 应用场景 | AI 可自动补，事后汇报 |
| `related` 相关节点 | AI 可自动补，低风险 |
| `chains` 能力链 | AI 可起草，但主链需要人确认 |
| `status` 掌握状态 | 必须人确认 |

核心规则：

> **状态和主链不能自动裁决，普通引用关系可以交给 AI 维护。**

---

## 2. 需要有“候选区”和“确认区”

AI 维护出来的关系不一定马上可信。

建议状态区分：

```yaml
status: candidate      # 候选
status: unstable       # 不稳定
status: confirmed      # 已确认
status: risk           # 带风险推进
```

再加一个关系确认字段：

```yaml
relation_status: candidate
```

或者更简单：第一版不加复杂字段，只规定：

> **AI 新增关系默认是候选关系，只有用户明确认可后才算稳定关系。**

---

## 3. 每次维护必须输出变更摘要

否则你会不知道 AI 改了什么。

每轮维护后必须汇报：

```text
本次新增了哪些节点
本次修改了哪些关系
本次标记了哪些风险
本次建议用户确认哪些判断
本次没有写入 baseline 的内容
```

这个比“自动维护成功”重要得多。

---

> **AI 可以维护结构，但不能静默改变学习路线；凡是影响前置关系、主能力链、掌握状态、baseline 的内容，都必须显式汇报并由用户裁决。**

这一条补上后，整个系统会更稳。
