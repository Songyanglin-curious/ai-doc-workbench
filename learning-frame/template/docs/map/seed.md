# Seed Map：工程能力候选起点

本文件是给你的第一批候选能力方向，不是已确认路线。使用时从真实问题出发，保留有用的，删除不贴近当前阶段的。

## 候选问题场景

### 接口联调失败

真实表现：页面请求失败、浏览器报错、Postman 可以但浏览器不行、线上环境和本地表现不同。

候选能力链：

```text
确认请求是否发出
-> 判断浏览器是否拦截：CORS / 预检 / mixed content
-> 判断网络是否连通：域名 / IP / 端口 / 服务启动
-> 判断服务端是否收到：日志 / 路由 / 方法 / 网关
-> 判断鉴权是否失败：token / header / cookie / session
-> 判断业务处理是否异常：参数 / SQL / 返回结构
-> 用最小复现收口
```

候选能力节点：

- 能判断一次接口请求失败卡在哪一层。
- 能区分 CORS 问题和服务端 404/500。
- 能用浏览器 DevTools、curl/Postman、服务端日志做分层排查。
- 能解释 header、cookie、token 在一次请求里分别承担什么角色。

### 登录态、鉴权、权限混乱

真实表现：登录成功但接口 401/403、退出登录后仍可访问、菜单有但接口无权限、数据越权。

候选能力链：

```text
区分身份凭证和登录态
-> 区分认证和授权
-> 区分菜单权限、接口权限、数据权限
-> 判断状态保存位置：token / session / Redis / DB
-> 验证失效方式和越权风险
```

候选能力节点：

- 能区分 token、session、Redis 登录态分别保存什么以及如何失效。
- 能区分认证失败、授权失败、数据权限失败。
- 能设计一个最小测试验证退出登录是否真的失效。

### 代码改造不知道影响范围

真实表现：能改一处，但担心影响其它模块；看懂代码片段，却不知道入口、调用链和测试点。

候选能力链：

```text
定位入口
-> 找调用链
-> 区分核心路径和边缘路径
-> 找状态和数据边界
-> 设计最小回归验证
-> 记录影响范围
```

候选能力节点：

- 能从一个报错或页面行为定位后端入口。
- 能沿调用链找到关键状态读写点。
- 能为一次小改动设计最小回归测试。

### 数据库读写和一致性问题

真实表现：查出来的数据不符合预期、并发后状态错乱、事务回滚和提交边界不清。

候选能力链：

```text
确认查询条件和数据来源
-> 判断事务边界
-> 判断并发读写影响
-> 判断缓存或异步流程是否参与
-> 用最小数据集复现
```

候选能力节点：

- 能解释一个业务操作里哪些 SQL 必须处于同一事务。
- 能用最小数据集复现一次状态异常。
- 能区分读错数据、写错数据、事务边界错、缓存未更新。

### AI 写出来我看懂，但自己改不动

真实表现：AI 生成代码能跑，但自己无法解释关键决策；换个条件就不会改。

候选能力链：

```text
要求 AI 说明设计选择
-> 自己复述核心流程
-> 删除代码后无 AI 重写关键部分
-> 做一个变式需求
-> 对失败点做前置回退
```

候选能力节点：

- 能把 AI 生成的代码拆成输入、状态、分支、输出。
- 能无 AI 重写核心逻辑的最小版本。
- 能对同一题做一个变式实现，而不是只复刻答案。

## 使用方式

每次只选一个候选场景，进入 `current/` 写当前台阶。

不要一次性把这里拆成几十个节点。只有当某个候选真的进入学习或验证，才把它提升为独立的 scenario、chain 或 capability-node 文件。
