# AGE Lite：从重模板到轻骨架的设计思路

AGE Lite 不是把 AGE 简单删薄一层，而是把原本偏重的治理模板，压缩成一套低摩擦、低上下文成本、又能继续长大的工作骨架。

它要解决的核心问题不是“目录太多”，而是 AI 容易一上来就读很多文档、扫很多代码、补很多礼貌性上下文，最后让小任务也像大项目治理。

## 一句话定位

- 强骨架，弱内部结构
- 输入驱动路由，不是固定阅读链
- 保留系统性治理能力，但不强迫每次都走完整治理链

## 吸引子思想

这套模板的核心不是把所有信息都装进文档，而是给项目保留一个稳定吸引子，让系统围绕少量已确认的真相持续收敛。

这里的吸引子不是“长篇目标文档”，而是下面这组约束：

- 用户当前输入优先
- live repo 决定当前实现实际如何工作
- `baseline/` 承载当前已确认的项目真相和长期收敛方向
- `plans/` 与 `logs/` 只记录执行轨迹，不反过来定义系统
- `input/` 和 `references/` 保留来源材料，但不伪装成项目结论

这意味着项目一开始可以不完整，也可以边做边收敛；但未确认的内容必须留在输入层或参考层，只有真正稳定下来的结构性判断，才进入 `baseline/`。

## 为什么要做目录简化

AGE 从重到轻，依据不是“越少越好”，而是只保留真正稳定、长期有价值的边界。

保留下来的，是 6 个角色明确、很难被替代的层：

- live repo
  - 真实代码和真实文件系统本体
- `docs/context/`
  - AI 如何进入项目、如何搜索、何时停下
- `docs/baseline/`
  - 当前真相与长期收敛方向
- `docs/plans/`
  - 复杂任务如何收口
- `docs/logs/`
  - 已落地事实
- `docs/input/` / `docs/references/`
  - 原始输入与外部约束

被拿掉或弱化的，是这些高摩擦习惯：

- 默认固定阅读链
- 每轮都读一圈 `context/baseline/plans/logs`
- 小任务默认补完整文档上下文
- 只要改动就自动写日志
- 预建大量目录，靠结构“看起来完整”

简化的标准只有一个：  
如果某一层不能稳定回答一种不同的问题，它就不该单独存在。

## 这套骨架真正固定的东西

### 1. 强骨架

顶层职责固定：

- `context` 解决“AI 怎么进来”
- `baseline` 解决“项目现在认可什么”
- `plans/logs` 解决“这一轮怎么收口、发生了什么”
- `input/references` 解决“来源材料是什么”

### 2. 弱内部结构

目录内部默认从简：

- 不预设大量子目录
- 先靠 `index.md` 做导航和局部说明
- 只有当内容真实增长到值得拆分时，再细分

这样做的目的，是让模板适配小项目、探索期项目和逐步长大的项目，而不是一开始就要求完整治理外观。

## 系统基础

这套系统的基础不是文档本身，而是三层现实分工：

### 1. 当前输入

用户这次到底要什么，始终优先于文档体系。  
如果用户已经给出明确文件、函数、报错、页面、样式或目录，默认直接去目标，而不是先回头读整套文档。

### 2. 当前真相

系统“应向哪里收敛”的问题，由 `baseline/` 回答。  
系统“现在实际怎么工作”的问题，由 live repo 回答。  
这两者不能混。

### 3. 当前轨迹

复杂任务如何收口，看 `plans/`。  
已经发生过什么，看 `logs/`。  
它们是轨迹层，不是定义层。

## 基础流转

这套模板的流转，不是从文档开始，而是从输入开始。

### 路径 1：明确小任务

- 读 `AGENTS.md`
- 直接读用户指定目标
- 直接处理
- 只有当 AI 判断这次结果值得维护，并准备实际写 `logs/`、索引或说明文档时，才发起维护确认

### 路径 2：模糊任务

- 读 `AGENTS.md`
- 进入 `docs/index.md`
- 按输入路由到 `input`、`references`、`baseline`、`plans` 或 `logs`
- 只有证据不足时，才进入 `context/`

### 路径 3：结构判断任务

- 读 `AGENTS.md`
- 进入 `docs/index.md`
- 转到 `baseline/`
- 若要判断当前实现行为，再回到 live repo

### 路径 4：复杂任务收口

- 读 `AGENTS.md`
- 进入 `docs/index.md`
- 转到 `plans/current.md`
- 在实现完成后决定是否需要维护

## 维护为什么也要变轻

轻量模板不等于“AI 自动维护一切”，也不等于“维护全靠人手工补”。

这里的原则是：

- 不是每个任务都触发维护
- 小修本身不触发维护
- 只有当 AI 判断这次结果值得写 `logs/`、补索引或更新说明时，才触发维护确认
- AI 先问“这次是否确认完成，是否允许我做维护”
- 用户给出明确信号后，AI 再执行维护
- 确认痕迹默认留在对话中，不额外生成状态文件

这样做的目的，是把维护从“每次改动都自动跟一串流程”改成“按需收口”。

## 这套模板不想变成什么

- 不想变成资料堆
- 不想变成固定阅读链
- 不想变成每轮都要治理的流程机器
- 不想让 `baseline/` 退化成材料索引
- 不想让 `plans/` 或 `logs/` 反过来污染系统定义

## 最终目标

这套精简不是为了更好看，而是为了同时做到三件事：

- 小任务足够快
- 大项目还能长
- AI 能维护，但不会越权

如果用一句话概括 AGE Lite：

它保留 AGE 的系统性治理思想，但把运行方式改造成输入驱动、低摩擦、可渐进生长的轻量骨架。
